> 下游 UI 设计说明（SC-*）与垂直切片均引用本文件的 ID。

术语：**权限码**=「资源:动作」形如 `sys:user:write`；**菜单**=权限载体（目录/菜单/按钮三类）；
**软删除**=逻辑删除（`is_deleted=1`），保留原始数据，绝不物理删除；
**信封加密**=在 TLS 之上，对敏感字段在请求/响应体内用国密再加密一层。

## 2 背景与产品定位

各业务系统都需要一套标准的「系统管理」后台：管账号、管角色权限、管菜单、查日志、维护字典。
本套餐把 Plumbline 已有的安全底座（国密加解密、脱敏、审计、登录链路、按钮级权限守卫）
组装成开箱即用的五大模块，并把原先**硬编码的角色权限**升级为**数据库驱动的动态 RBAC**。

- **范围 In**：用户管理、角色管理、菜单管理、日志管理、字典管理；水平/垂直越权加固；敏感信息加密存储、脱敏展示、传输信封加密。
- **范围 Out**：组织机构/部门树、工作流审批、单点登录（SSO）、多租户（后续迭代）。

## 3 角色与权限模型

动态 RBAC：用户—角色—菜单（权限）三层。角色与菜单落库可维护，权限即数据。

| 概念 | 落点 | 说明 |
|------|------|------|
| 用户 | `User` 表（既有，扩展软删/审计字段） | `roles_csv` 存角色 code |
| 角色 | `Role` 表（新建） | code 唯一，可增改软删 |
| 菜单/权限 | `Menu` 表（新建） | 树形，`type∈{dir,menu,button}`，button 的 `perm_code` 即按钮级权限码 |
| 角色-菜单 | `RoleMenu` 关联（新建） | 角色被授予哪些菜单/权限 |

内置角色（seed 初值，可在后台维护）：`管理员`(全权)、`评审专家`、`回测分析员`、`标准配置员`。

## 4 总体方案设计

| 层 | 职责 |
|----|------|
| 应用层 | 五个管理工作台（用户/角色/菜单/日志/字典） |
| 守卫层 | `require_perms` 从 DB（Redis 缓存）解析角色→权限码；数据范围过滤防水平越权 |
| 安全层 | 国密加解密、脱敏、传输信封、审计留痕（复用 `core/`） |
| 数据层 | 系统表统一软删除 + 审计字段四件套 |

核心原则：权限即数据、越权双向防护、敏感信息全链路保护、删除可逆、写操作必留痕。

## 5 数据需求与数据模型

所有系统表统一继承公共基类，带软删除标记与审计字段四件套。

| 实体 | 关键字段/关系 |
|------|----------------|
| 公共基类 SysBase | `created_at`、`updated_at`、`created_by`、`updated_by`、`is_deleted` |
| User（扩展） | id、username、password_hash、phone_enc、email_enc、roles_csv、status、+SysBase |
| Role | id、code(唯一)、name、remark、+SysBase |
| Menu | id、code(唯一)、name、parent_id、type、perm_code、path、icon、order、+SysBase |
| RoleMenu | role_id、menu_id |
| Dict / DictItem | 字典类型 code/name；字典项 type_code、label、value、order、status |
| AuditRecord / LoginLog | 既有（`core/audit`）：操作日志、登录日志，只增不改 |

## 6 功能需求详述

### 6.1 用户管理

模块目标：管理系统账号，敏感信息加密存储、脱敏展示、传输信封保护。

- **6.1.1** 分页/按用户名搜索查询用户列表，手机号/邮箱脱敏展示。`[FR-6.1.1]` `[NFR-6.6]`
- **6.1.2** 新建用户：口令 PBKDF2-SM3 哈希；手机号/邮箱 SM4 加密落库，入参经传输信封解密。`[FR-6.1.2]` `[NFR-6.5]` `[NFR-6.7]`
- **6.1.3** 编辑用户基本信息（敏感字段同样加密/信封处理）。`[FR-6.1.3]`
- **6.1.4** 重置/修改用户口令，符合口令策略。`[FR-6.1.4]`
- **6.1.5** 启用/停用用户（`status` 切换），停用即不可登录。`[FR-6.1.5]`
- **6.1.6** 软删除用户：置 `is_deleted`，保留原始数据，列表默认不展示。`[FR-6.1.6]` `[NFR-6.1]`
- **6.1.7** 给用户分配角色（写 `roles_csv`），变更即影响其权限。`[FR-6.1.7]`

### 6.2 角色管理

模块目标：维护角色与其菜单/权限授权，驱动动态 RBAC。

- **6.2.1** 分页查询角色列表。`[FR-6.2.1]`
- **6.2.2** 新建/编辑角色（code、name、备注）。`[FR-6.2.2]`
- **6.2.3** 软删除角色（保留原始数据；已被用户引用时给出提示）。`[FR-6.2.3]` `[NFR-6.1]`
- **6.2.4** 给角色分配菜单/按钮权限（写 `RoleMenu`），保存后即时失效权限缓存。`[FR-6.2.4]` `[NFR-6.3]`

### 6.3 菜单管理

模块目标：维护菜单树与按钮级权限码，作为 RBAC 的权限载体。

- **6.3.1** 树形展示菜单（目录/菜单/按钮三级）。`[FR-6.3.1]`
- **6.3.2** 新建/编辑菜单节点（name、path、icon、order、parent）。`[FR-6.3.2]`
- **6.3.3** 软删除菜单节点（含子节点级联标记）。`[FR-6.3.3]` `[NFR-6.1]`
- **6.3.4** 维护按钮节点的权限码 `perm_code`（如 `sys:user:write`）。`[FR-6.3.4]` `[NFR-6.3]`

### 6.4 日志管理

模块目标：可查询的操作日志与登录日志，敏感信息脱敏。

- **6.4.1** 按实体/操作人/时间范围分页查询操作日志（`AuditRecord`）。`[FR-6.4.1]` `[NFR-6.4]`
- **6.4.2** 按用户名/结果/时间范围分页查询登录日志（`LoginLog`）。`[FR-6.4.2]`
- **6.4.3** 日志中的敏感信息（手机号等）脱敏展示，不落明文。`[FR-6.4.3]` `[NFR-6.6]`

### 6.5 字典管理

模块目标：维护可复用的数据字典（类型 + 字典项）。

- **6.5.1** 字典类型增删改查（软删）。`[FR-6.5.1]` `[NFR-6.1]`
- **6.5.2** 字典项增删改查（隶属某类型，软删）。`[FR-6.5.2]`
- **6.5.3** 按字典类型 code 拉取启用中的字典项（供前端下拉等使用）。`[FR-6.5.3]`

功能模块与优先级：6.1/6.2/6.4（P0）；6.3/6.5（P1）。

| 维度 | 要求 | ID |

| 软删除 | 系统表逻辑删除，保留原始数据，绝不物理删 | `[NFR-6.1]` |
| 审计字段 | 每张系统表带 created_at/updated_at/created_by/updated_by | `[NFR-6.2]` |
| 动态 RBAC | 角色/菜单/权限落库，权限即数据，缓存可即时失效 | `[NFR-6.3]` |

| 传输加密 | TLS 之上，敏感字段在请求/响应体内国密信封加密 | `[NFR-6.7]` |

## 8 接口需求（概要）

- 用户/角色/菜单/字典 CRUD：`/api/sys/*`，写操作 `require_perms("sys:<资源>:write")`，软删走 PATCH/DELETE 但仅置标记。
- 日志查询：`/api/sys/logs/audit`、`/api/sys/logs/login`，只读，`require_perms("sys:log:read")`。
- 传输信封：`GET /api/crypto/pubkey` 下发 SM2 公钥；敏感接口入参解信封、出参脱敏。
- 所有业务接口带 `operation_id` 与 `openapi_extra={"x-trace":[FR-6.*]}`。

## 9 验收标准与度量指标

- 无权限角色访问写接口返回 403；越权访问他人数据范围返回 403/空（回指 `[NFR-6.4]`）。
- 删除后数据库记录仍在、仅 `is_deleted=1`，列表不展示（回指 `[FR-6.1.6]` `[NFR-6.1]`）。
- 手机号/邮箱在库为 SM4 密文、接口出参为掩码（回指 `[NFR-6.5]` `[NFR-6.6]`）。
- 角色授权变更后，权限缓存即时失效、守卫按新权限放行/拒绝（回指 `[FR-6.2.4]` `[NFR-6.3]`）。

## 12 附录

权限码命名规范：`sys:<资源>:<动作>`，资源 ∈ {user, role, menu, dict, log}，动作 ∈ {read, write}。
例：`sys:user:read`、`sys:user:write`、`sys:role:write`、`sys:log:read`。
