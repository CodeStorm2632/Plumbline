# 标准 PRD 大纲、追溯 ID 约定与写作要点

这是 `prd-writer` 的标准骨架。**按需裁剪**章节（小项目可合并 6/8、去掉 8），但尽量保留 PM 必备结构。功能需求（第 5 章）是主体。

---

## 〇、追溯 ID 约定（先看，这是整条流水线的脊柱）

PRD 给三类对象分配稳定 ID，下游 `prd-to-ui-spec` / `spec-to-slice` 全靠它们追溯。**权威版本在 `dev-pipeline` 总纲，这里是内联速记。**

| 类型 | 形态 | 编号规则 | 例 |
|------|------|----------|----|
| 功能需求 | `[FR-<模块>.<序>]` | 跟 §5 模块号；模块内顺序递增 | `[FR-5.1.3]` = 5.1 模块第 3 条 |
| 非功能需求 | `[NFR-<序>]` | 全局顺序 | `[NFR-4]` = 审计留痕 |
| 开放问题 | `[OQ-<序>]` | 全局顺序，§11 定义 | `[OQ-2]` |

规则：
- 一条 FR 只描述**一件可独立验证的事**。三件事拆三条，别塞进一条。
- ID 写在该需求条目的句首或句末，方括号包裹（正则 `\[(?:FR|NFR|OQ)-[0-9.]+\]` 能命中）。
- **只增不改、不重排**：修订只往后追号；删除的需求标注废弃但不回收号，避免下游 trace 失效。
- 下游只能**引用** PRD 已有 ID，不得新增需求；缺需求回本 skill 补。

## 〇.一、Frontmatter（必带）

PRD.md 顶部带 YAML frontmatter，下游据此识别版本与技术栈：

```yaml
---
id: PRD-<slug>                # 全局唯一，如 PRD-talent-eval
title: <系统名> 产品需求说明书
version: v1.0                  # vMAJOR.MINOR；ID 变更与版本绑定
status: draft                 # draft | review | approved
stack: { framework: React19+TS, router: TanStack Router, data: TanStack Query, table: TanStack Table, ui: Shadcn/UI, css: Tailwind4, schema: Zod4, backend: FastAPI+SQLModel+Postgres/Alembic+Redis }
---
```

`stack` 是技术栈的**唯一声明处**，立项时与用户确认；它会一路流到 UI 说明与切片模板。

## 二、逐章要点

### 1 文档说明
- 修订记录表：版本 / 日期 / 修订人 / 说明。版本与 frontmatter `version` 一致。
- 术语表：领域黑话、缩写、关键概念定义清楚，全文统一口径。

### 2 背景与产品定位
- **业务背景**：现状如何做、量级多大、痛点是什么（效率/一致性/可追溯/可维护）。
- **产品定位**：一两句可记忆的定位。常用「X 的智能助手而非替代者」明确人机分工。
- **产品价值**：效率/一致/可解释/可演进，分条且每条给实质。
- **范围**：In Scope 编号列出本期能力；Out of Scope 明确划走，避免无边界扩张。

### 3 目标用户与典型场景
- 角色表：角色 | 职责 | 对系统的核心诉求。覆盖配置者、操作者、复核者、审核者、管理员等。**下游屏清单按角色保证每类角色都有界面。**
- 端到端流程：编号步骤串起「配置→导入→处理→复核→校验→迭代」主链路。

### 4 总体方案设计
- 分层/架构表：每层职责一句话。**「应用层」列出的工作台/后台（配置台、评价台、回测台、审计后台…）是下游屏清单的直接推导源——务必写清。**
- 核心设计原则：如「配置即标准」「计算可解释」「自动负责量、人工负责判」「版本可回溯」，并讲清「为什么重要」。

### 5 功能需求详述（主体，带 FR）
- 按模块拆 5.1 / 5.2 …；每模块：
  - 模块目标（一句话）。
  - 关键能力/规则：**逐条写，每条挂 `[FR-5.x.y]`**。复杂规则用表格（规则类型 | 说明 | 样例）或代码块表达。
  - 边界与约束、异常处理。
  - 可解释/可追溯要求（如涉及自动计算）。
- 示例写法：
  > **5.1.3** 系统对每个申报人输出可解释计分明细（指标—规则—证据—计算式—佐证—材料定位）。`[FR-5.1.3]`
- 末尾给「功能模块与优先级汇总」表（模块 | FR 范围 | P0/P1 | 说明）。

### 6 数据需求与数据模型
- 核心实体表：实体 | 关键字段/关系。逻辑模型即可，物理设计另行。字段命名与 indicators.yaml、后端 DTO 对齐。

### 7 非功能需求（带 NFR）
- 维度 | 要求 | ID 表：可解释性、准确性、安全/密级、审计、性能、可用性、可演进、合规，每条挂 `[NFR-n]`。结合行业约束（涉密、数据分级）。
- 常用固定语义：`[NFR-3]` 密级管控、`[NFR-4]` 操作留痕/审计——下游接口 x-trace 常引用，尽量沿用。

### 8 接口需求（概要）
- 数据导入、结果导出、外部服务（如大模型）调用；强调留痕/可审计。

### 9 验收标准与度量指标
- 功能验收：编号的可验证清单，**回指对应 `[FR-*]`**。
- 量化度量表：指标 | 定义 | 用途。阈值留给业务方首测后设定，不预设硬指标。

### 10 里程碑与迭代计划
- 阶段 | 目标 | 交付物。MVP 跑通最小闭环，V1/V2 补齐校验与扩展。建议标注每阶段覆盖的 FR 范围。

### 11 风险与开放问题（务必单列，带 OQ）
- 每个开放问题一小节并挂 `[OQ-n]`：写清现状、冲突点、需要谁在什么时点拍板。
- 用醒目色标注「待明确」（`**[OQ-1] 待明确** …`）。常见来源：口径/权重冲突、判定规则待定、阈值待标定、数据质量、外部模型可靠性与合规。
- **不替用户拍板有争议的口径。**

### 12 附录（强烈建议）
- 把参考标准原样整理成表格：指标/字段/规则/负面清单/上限/排名系数等。
- 这是「文档可落地」的关键，也是 **indicators.yaml** 的直接素材（schema 见 `indicators-schema.md`）。

- [ ] §5 每个叶子模块至少 1 条 `[FR-*]`，每条只描述一件可验证的事。
- [ ] 所有 ID 唯一、不重复、不重排。
- [ ] In/Out Scope 都写了。
- [ ] 所有「待定」都进了 §11 并挂 `[OQ-*]`、标注决策人/时点。

- [ ] 有 MVP→V1→V2 里程碑。
- [ ] 来源标准映射进了 §12 附录（必要时导出 indicators.yaml）。
